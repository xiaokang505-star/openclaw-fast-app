# OpenClaw 安装/配置助手 — 项目分析与规划

## 一、项目目标

在电脑上通过一个 **Electron 桌面应用**，帮助**所有用户**（包括零环境用户）**快速安装和配置 OpenClaw**。应用需：

- 覆盖从「什么都没有」到「可正常使用」的**每一步**安装与配置；
- 在**每一步**检测用户当前环境，只做必要操作、避免重复；
- 根据**已有环境**自动跳过已完成步骤，从当前状态继续完成剩余搭建。

### 1.1 应用整体流程：三大阶段

本应用在流程上分为**三大阶段**，顺序进行；完成后用户进入**主界面（Agent 对话）**日常使用。设置入口在**安装阶段**与**主界面**均可进入，无需重新走完整向导。

---

#### 阶段一：安装阶段

- **目标**：将运行 OpenClaw 所需的**所有依赖**安装完毕（Node/npm、OpenClaw CLI、Daemon），并**询问用户是否需要免费本地模型支持**；若需要，则安装并配置 **electron-ollama** 套件。
- **流程**：
  1. **功能检测**：进入安装阶段时先执行功能检测（见 6.0），得到当前环境状态与未通过步骤。
  2. **按依赖链安装**：按步骤 0～5（运行环境 → Node → npm → OpenClaw CLI → Daemon/网关 → 配置文件存在性）逐项检测并执行安装；**安装过程需展示安装进度**（如 npm 安装日志、Daemon 启动状态）。步骤 4 若「安装并启动 Daemon」报错，提供**由本应用启动网关**（spawn `openclaw gateway`，兼容性更好；关闭应用后网关停止）。
  3. **免费模型询问**：在安装步骤中或步骤 5 前，**询问用户是否启用免费本地模型**；若用户选择「是」，则进入步骤 5b，安装/启动 **electron-ollama**，引导选择或拉取模型，并通过 Ollama 适配器写入 OpenClaw 配置。
  4. **设置入口**：安装阶段界面**右上角常驻「设置」按钮**，用户可随时进入设置菜单切换 npm 源、Node 路径、nvm 镜像等，无需退出向导；设置与安装共用同一套检测与执行逻辑。

**产出**：Node、npm、OpenClaw CLI、Daemon 就绪；可选 Ollama 就绪；`~/.openclaw/openclaw.json` 已存在（可为空或含 Ollama provider）。完成后进入**阶段二**。

---

#### 阶段二：OpenClaw 配置阶段

- **目标**：**补全当前方案漏洞**——在「安装」之后、进入主界面前，**图形化列出 OpenClaw 的基础配置项**，让用户选择或填写，写回 `openclaw.json`，再退出安装向导。
- **流程**：
  1. **图形化配置项**：在应用内以表单/卡片形式列出 OpenClaw 的**基础配置项**（见下「1.2 OpenClaw 基础配置项」），例如：模型来源（云端 API / 本地 Ollama）、API Key（若选云端）、默认模型、网关端口等；用户逐项选择或填写。
  2. **校验与写回**：用户确认后，将表单结果合并写入 `~/.openclaw/openclaw.json`（注意 JSON5 格式与已有字段合并，避免覆盖用户已有配置）。
  3. **完成向导**：用户点击「完成配置」或「退出安装向导」后，**安装向导结束**，应用界面切换为**阶段三：主界面（Agent 对话）**。

**产出**：`openclaw.json` 中基础项已按用户选择填写完毕；用户明确「配置已完成」，进入日常使用界面。

---

#### 阶段三：主界面（Agent 对话）

- **目标**：安装与配置全部完成后，应用**主界面**呈现 **Agent 对话样式**，供用户与 OpenClaw 网关对话（可操作本机）。
- **对话与布局**：
  - **对话后端**：固定使用 **OpenClaw 网关**（`/v1/chat/completions`），不提供 Ollama/OpenClaw 切换；需 Daemon 已启动且配置中已启用 `gateway.http.endpoints.chatCompletions.enabled`。
  - **回复展示**：助手回复支持 **Markdown 渲染**（代码块、列表、标题等），使用 marked + DOMPurify 安全输出。
  - **布局**：**输入框一行固定在底部**，不随对话内容多少而移动；中间为可滚动的消息区。
- **界面形态**：
  - **上方**：AI 对话内容展示区（消息列表、角色区分、流式输出、Markdown 渲染）。
  - **底部固定**：**输入框** + **发送按钮** + **「设置」按钮**；设置入口与阶段一一致。
- **与设置联动**：主界面修改设置后，若影响 OpenClaw 运行（如重装 openclaw、更换模型），需有明确提示或「重新检测」入口。若对话出现 **HTTP 404**，错误提示会引导用户确认 Daemon 已启动且 Chat Completions 端点已启用；设置中提供「一键应用本地模型」可写入 Ollama 并设为默认，减少 404。

---

#### 设置菜单（贯穿阶段一与阶段三）

- **入口**：**阶段一**安装界面右上角「设置」；**阶段三**主界面输入框旁（发送按钮旁）「设置」。主界面与设置页均可进入 **OpenClaw 配置**（阶段二表单），无需重新走安装向导。
- **必备项**：Node 环境地址、npm 源地址、nvm Node 镜像（若有）、**OpenClaw 基础配置**（打开配置页）、**一键应用本地模型**（选择 Ollama 模型后写入配置并设为默认，可避免对话 404）、OpenClaw 包（版本/重新安装/打开配置目录）、**由本应用启动网关**（推荐，兼容性高）/ 停止网关 / 安装并启动 Daemon；可选诊断日志等。
- **安装向导仅首次可见**：完成过一次安装向导后（`wizardCompleted`），再次打开应用直接进入主界面；从设置点击「返回」时进入**主界面**而非带「开始安装」的首页，避免歧义。设置页对已完成用户展示说明：「安装向导仅在首次使用时展示；之后可在此管理全部安装与配置项。」
- **原则**：与安装阶段共用同一套环境检测与执行逻辑；修改后引导或主界面中的后续操作使用最新配置。

**进入安装阶段前的功能检测**：用户点击「开始安装」或首次进入安装阶段时，先执行**功能检测环节**（6.0），再根据检测结果决定是否进入、以及从哪一步开始（见第六节 6.0）。

---

### 1.2 OpenClaw 基础配置项（阶段二图形化列出）

阶段二「OpenClaw 配置」中，需在应用内**图形化列出**并让用户选择/填写的**基础配置项**（对应 `~/.openclaw/openclaw.json` 的常用字段，具体以 OpenClaw 官方文档为准）：

| 配置项 | 说明 | 交互方式 |
|--------|------|----------|
| **模型来源** | 云端 API / 本地 Ollama | 单选：云端 / 本地（若已装 Ollama 或 electron-ollama） |
| **API Key / 端点**（若选云端） | 云端模型 API Key、Base URL（可选） | 输入框；可选「从环境变量读取」 |
| **默认模型** | 使用的模型 ID 或名称 | 下拉（从当前 provider 的 models 列表拉取）或输入 |
| **本地模型**（若选本地） | 已选 Ollama 模型、拉取新模型 | 模型列表选择；「拉取新模型」按钮（步骤 5b 可能已做，此处可只做展示/切换） |
| **网关端口**（可选） | Daemon 监听端口，默认如 18789 | 数字输入；端口冲突时提示 |
| **其它必要项** | 以 OpenClaw 文档为准 | 表单或「高级：打开配置文件」 |

实现时：读取现有 `openclaw.json`（若存在）预填表单；用户提交后**合并写回**，不覆盖未在表单中编辑的其它键；支持「打开配置目录」由用户手动编辑。

## 二、OpenClaw 背景（调研摘要）

根据公开文档整理：

1. **OpenClaw 是什么？**
   - 开源 AI 编程助手，以 **CLI + 本地网关/服务** 形式运行。
   - 官方推荐安装：`npm install -g openclaw@latest`，然后 `openclaw onboard --install-daemon`。
   - 文档参考：<https://docs.openclaw.ai/setup>、<https://learnopenclaw.com/getting-started/installation>。

2. **“安装”具体指什么？**
   - 通过 **npm 全局安装** CLI，再通过 `onboard --install-daemon` 安装/启动本地守护进程。
   - 前置条件：**Node.js 24+**（推荐 LTS）、8GB+ 内存、10GB+ 磁盘等。
   - 我们的 Electron 应用可以：检测 Node 版本 → 引导执行上述命令 → 显示进度与结果。

3. **“配置”具体指什么？** 
   - 配置文件：**`~/.openclaw/openclaw.json`**（JSON5 格式）。
   - 方式：交互式向导 `openclaw onboard` / `openclaw configure`、CLI `config get/set/unset`、或控制台 UI `http://127.0.0.1:18789`。
   - 我们可做：在应用内提供表单编辑该文件、或一键打开控制台/启动 onboard。

4. **目标用户与平台**
   - 官方支持：Windows 10+（推荐 WSL2）、macOS 12+、Ubuntu 20.04+。
   - 可先做 **macOS + Windows**，再考虑 Linux。

5. **免费本地模型（Ollama）**
   - OpenClaw 支持通过 **OpenAI 兼容** 的 endpoint 接入本地模型；Ollama 提供 `http://localhost:11434/v1` 的兼容接口。
   - 我们可在应用内集成 **electron-ollama**，为用户提供「不依赖云端 API、完全免费」的 OpenClaw 使用方案。

## 三、多平台与多架构适配：环境异同与规划信息

目标平台为 **Windows / macOS / Linux**，且需覆盖各主流**架构**（x64、arm64 等）。下面罗列各平台×架构的**环境异同**，以及规划/实现时**需要包含的信息**。

### 3.A 支持矩阵（OS × 架构）

| 操作系统 | 常见架构 | Node 官方包 | 说明 |
|----------|----------|-------------|------|
| **Windows** | x64、arm64 | win-x64 / win-arm64（.msi 或 .zip） | 另有 x86；arm64 多为 ARM 笔记本/平板 |
| **macOS** | x64（Intel）、arm64（Apple Silicon） | darwin-x64 / darwin-arm64（.pkg 或 .tar.gz） | Apple Silicon 为当前主流，需单独检测 |
| **Linux** | x64、arm64、armv7l 等 | linux-x64 / linux-arm64 等（.tar.xz） | 发行版众多，PATH 与包管理器不一 |

检测时建议使用 Node/Electron 的 `process.platform`（`win32`/`darwin`/`linux`）与 `process.arch`（`x64`/`arm64` 等）确定「当前运行环境」，以便选择正确的安装包与命令。

### 3.B 各平台环境异同总表

| 项目 | Windows | macOS | Linux |
|------|---------|--------|--------|
| **Node 典型安装位置** | 用户选择或 `%ProgramFiles%\nodejs\`、nvm 等 | `/usr/local`、nvm/fnm 的 `~/.nvm` 等 | `/usr/local`、nvm/fnm、系统包管理器 |
| **npm 全局 prefix 默认** | `%AppData%\npm`（如 `C:\Users\<用户>\AppData\Roaming\npm`） | `/usr/local` | `/usr/local` |
| **npm 全局包路径** | `%AppData%\npm\node_modules`，可执行文件在同级或 `node_modules\.bin` | `/usr/local/lib/node_modules`，bin 在 `/usr/local/bin` | 同 macOS；或 `~/.npm-global` 等自定义 |
| **用户主目录（homedir）** | `%USERPROFILE%`（如 `C:\Users\<用户>`） | `$HOME`（如 `/Users/<用户>`） | `$HOME`（如 `/home/<用户>`） |
| **OpenClaw 配置目录** | `%USERPROFILE%\.openclaw\`，配置文件 `openclaw.json` | `~/.openclaw/openclaw.json` | `~/.openclaw/openclaw.json`；可用 `OPENCLAW_STATE_DIR` 覆盖 |
| **路径分隔符** | `\`（Node 中通常也接受 `/`） | `/` | `/` |
| **PATH 生效时机** | 安装 Node 后有时需**重启终端或应用**才能拿到新 PATH | 新开终端即生效；GUI 应用可能需重启 | 同 macOS |
| **推荐安装 Node 的方式** | 官方 .msi、winget、choco、nvm-windows | 官方 .pkg、Homebrew、nvm/fnm | 官方二进制解压、nvm/fnm、apt/dnf 等 |
| **权限与提权** | 安装到 ProgramFiles 需管理员；用户目录不需 | 安装到 `/usr/local` 可能需 sudo；用户目录不需 | 系统级安装常需 sudo；用户目录不需 |
| **electron-ollama 二进制** | 按 `{basePath}/{version}/win32/{arch}/` 放置 | 按 `{basePath}/{version}/darwin/{arch}/` | 按 `{basePath}/{version}/linux/{arch}/` |

**共同点**：Node 24+、npm 随 Node、OpenClaw 通过 npm -g 安装、配置用 `~/.openclaw`（Windows 下即 `%USERPROFILE%\.openclaw`）；Ollama API 均为 `http://127.0.0.1:11434/v1`。

**差异要点**：路径与 homedir 解析（需用 `os.homedir()` 等跨平台 API）；Windows 下 PATH 刷新与可执行名（如 `openclaw.cmd`）；Linux 发行版与包管理器多样，需兼容「无 Node」与「有 Node 但非标准路径」两种情况。

### 3.C 各架构需注意的点

| 架构 | Windows | macOS | Linux |
|------|---------|--------|--------|
| **x64** | 主流，官方 .msi 直接支持 | Intel Mac，仍常见 | 最常见，官方 tar.xz 支持 |
| **arm64** | ARM 笔记本（如骁龙本）需 win-arm64 包 | **Apple Silicon（M1/M2/M3 等）**，必须用 darwin-arm64，与 x64 不通用 | 树莓派 4/5、部分服务器，用 linux-arm64 |
| **其他** | x86 可选支持 | — | armv7l 等可选；Power/ s390x 等按需 |

规划时需明确：**检测到 arm64 时，引导下载/安装的 Node、Ollama 等均为对应架构版本**，避免在 ARM 设备上误用 x64 二进制。

### 3.D 规划时需包含的信息清单

在设计与实现各步骤（环境检测、安装、配置、风险恢复）时，建议**每条能力/每步**都覆盖以下信息，以便正确适配 Win/Mac/Linux 及各架构：

1. **检测逻辑**
   - 当前 `platform`、`arch` 如何获取（如 `process.platform`、`process.arch`）。
   - Node/npm：用哪个可执行名（`node`/`node.exe`）、在何种 PATH 下执行（系统 PATH 还是应用内记录的 Node 路径）。
   - 配置文件与 homedir：是否统一用 `os.homedir()` 与 `path.join()`，Windows 下 `\.openclaw\` 与 Unix 下 `/.openclaw/` 的拼接是否正确。

2. **安装与下载**
   - 各平台×架构对应的**官方 Node 下载 URL**（或包管理器命令），以及是否提供「更换 npm 源」后的安装方式（如 npm 镜像站）。
   - electron-ollama 的二进制目录规则（`os/arch`）与是否支持当前 (platform, arch) 组合。
   - 安装完成后是否需要提示「请重启本应用」或「请新开终端」以使 PATH 生效（尤其 Windows）。

3. **路径与配置**
   - OpenClaw 配置路径：统一为 `path.join(os.homedir(), '.openclaw', 'openclaw.json')`；若支持 `OPENCLAW_STATE_DIR`，则需读取环境变量（子进程或用户设置）。
   - npm 配置：用户级 `.npmrc` 路径（`path.join(os.homedir(), '.npmrc')`），以及 `npm config get/set registry` 在各平台下的行为一致。

4. **权限与失败处理**
   - 哪些操作可能触发「需要管理员/ sudo」：例如 Windows 下安装到 ProgramFiles、Linux 下安装到 /usr/local。
   - 权限不足或路径不可写时的提示与降级方案（如改为用户目录安装）。

5. **UI 与文案**
   - 各步骤的**平台/架构相关提示**：例如「当前为 Apple Silicon，将使用 arm64 版本」「在 Windows 上安装 Node 后请重启本应用」。
   - 设置页中「Node 路径」「npm 源」等在不同平台下的展示与占位符（如 Windows 路径示例 vs Unix 路径示例）。

6. **测试与兼容**
   - 需在**真实环境**中验证的组合：至少 Windows (x64/arm64)、macOS (x64/arm64)、Linux (x64，可选 arm64)；若有 WSL2 场景，可单独注明「在 Windows 本机检测 Node，而非 WSL 内」。

将上述信息纳入需求与实现文档（如步骤说明、配置项说明、适配器规格），可保证各平台×架构行为一致且可维护。

## 四、免费本地模型方案（electron-ollama + Ollama 适配器）

### 4.1 目标

为希望**零成本、纯本地**使用 OpenClaw 的用户提供一条龙方案：

- 在 Electron 应用内通过 **electron-ollama** 管理/启动 Ollama，无需用户单独安装 Ollama；
- 让用户**选择并拉取**本地模型（如 llama2、qwen2、deepseek-coder 等）；
- 通过 **Ollama 适配器** 将 OpenClaw 的网关配置为使用本地 Ollama，实现完全免费的 OpenClaw 使用。

### 4.2 electron-ollama 集成

- **能力**：在 Electron 内嵌 Ollama 运行时，自动检测是否已有 Ollama 进程，若无则从指定版本启动（如 `v0.11.0`），避免与用户本机已安装的 Ollama 冲突。
- **典型用法**（主进程）：
  ```ts
  import { ElectronOllama } from 'electron-ollama'
  const eo = new ElectronOllama({ basePath: app.getPath('userData') })
  if (!(await eo.isRunning())) {
    await eo.serve('v0.11.0', { log: (msg) => console.log('[Ollama]', msg) })
  }
  ```
- **我们需做**：在「免费本地模型」流程中按需启动/检测 Ollama，并在 UI 中展示状态（运行中 / 未安装 / 启动中）。

### 4.3 用户选择模型

- 通过 Ollama 的 **API**（如 `GET /api/tags`）获取本地已拉取的模型列表；可选：展示推荐模型列表（如 coding 向的 `deepseek-coder`、`qwen2.5-coder`），支持「拉取」未安装的模型（调用 `ollama pull <model>`）。
- 在应用内提供**模型选择页**：列表展示 + 选择默认模型 + 可选「拉取新模型」入口，选定后作为 OpenClaw 的默认/本地模型。

### 4.4 Ollama 适配器（让 OpenClaw 连接本地 Ollama）

- **原理**：Ollama 提供 OpenAI 兼容的 Chat Completions 接口（`http://127.0.0.1:11434/v1`），OpenClaw 的网关支持配置「自定义 OpenAI 兼容」的 provider。
- **适配器职责**：
  1. **确保 Ollama 服务可用**：由 electron-ollama 在本应用内启动，或检测用户本机 Ollama 是否在 `localhost:11434` 监听。
  2. **写 OpenClaw 配置**：在 `~/.openclaw/openclaw.json` 的 `models.providers` 中增加一个本地 provider，例如：
     - `baseUrl`: `http://127.0.0.1:11434/v1`
     - `api`: `openai`
     - `models`: 将用户选择的模型（如 `llama2`、`qwen2.5-coder`）以 OpenClaw 所需的模型项格式写入（含 `id`、`name`、`reasoning`、`contextWindow`、`maxTokens` 等，成本可设为 0）。
  3. **可选**：若 OpenClaw 支持「仅用本地模型」模式，可将该 provider 设为首选或唯一，实现完全离线/免费。
- **实现形式**：在应用内提供「Ollama 适配器」模块（主进程或独立脚本）：读取当前选中的 Ollama 模型列表与默认模型 → 生成/合并上述 provider 配置 → 写回 `openclaw.json`，并提示用户重启或重载 OpenClaw 网关。

### 4.5 与主流程的衔接

- 在**配置步骤（步骤 5）**中增加**模型来源**选择：
  - **云端 API**：沿用现有流程（填写 API Key、选择模型等）。
  - **免费本地模型**：进入「本地 Ollama」子流程：
    1. 检测/启动 electron-ollama（或本机 Ollama）；
    2. 展示模型列表，用户选择或拉取模型；
    3. 调用 Ollama 适配器，写入 OpenClaw 配置并指向所选模型；
    4. 引导用户验证（如 `openclaw doctor` 或在 Cursor 中测一次）。
- 这样用户可在同一应用内完成「安装 OpenClaw → 选择用云端还是本地 → 若选本地则一键完成 Ollama 启动 + 选模型 + 写配置」，实现完全免费的 OpenClaw 使用方案。

## 六、技术方案：全流程分步检测与按需执行

### 6.0 进入安装引导前的功能检测环节

在用户**进入安装引导模式**之前，先执行一轮**功能检测**，不执行任何安装或写配置操作，仅收集环境状态并产出「检测报告」，用于决定是否允许进入引导、以及引导从哪一步开始展示。

#### 功能检测需执行的操作（按建议顺序）

| 序号 | 检测项 | 操作说明 | 产出/用途 |
|------|--------|----------|-----------|
| 1 | **运行环境** | 读取 `process.platform`、`process.arch`；可选：`os.totalmem()`/`os.freemem()`、磁盘剩余空间（如通过 Node 或系统调用） | 判断是否支持当前 OS/架构；若不支持则提示并**不进入**安装引导；记录 arch 供后续下载对应 Node/Ollama 包 |
| 2 | **Node 环境** | 使用「设置」中保存的 Node 路径（若有），否则用系统 PATH；在子进程执行 `node -v`，解析版本号；可选：`node -p process.execPath` 得到可执行路径 | 得到 Node 是否存在、版本是否 ≥24、实际使用的路径；若无或版本不符，引导将从「安装/升级 Node」开始 |
| 3 | **npm 环境** | 在同上环境中执行 `npm -v`；执行 `npm config get registry` 得到当前源地址 | 得到 npm 是否可用、版本、当前 registry；若不可用则引导需先解决 Node；registry 用于展示或后续安装时使用 |
| 4 | **OpenClaw CLI** | 在同上环境（确保 PATH 含 Node 全局 bin）执行 `openclaw -v` 或 `openclaw --version`，解析版本；若命令不存在则记「未安装」 | 得到 OpenClaw 是否已全局安装、版本；未安装则引导从「安装 OpenClaw」开始 |
| 5 | **OpenClaw Daemon / 网关** | 执行 `openclaw doctor` 或请求 `http://127.0.0.1:18789`（或 OpenClaw 文档中的网关端口）探测是否可达 | 得到 Daemon 是否已安装并运行；未运行则引导需执行 onboard --install-daemon 或启动服务 |
| 6 | **OpenClaw 配置文件** | 使用 `path.join(os.homedir(), '.openclaw', 'openclaw.json')` 读文件是否存在、是否可读；若存在则解析 JSON/JSON5，检查是否有「模型 / 网关」等必要字段（按 OpenClaw 文档定义） | 得到配置是否存在、是否已做基础配置；缺失或不完整则引导进入「配置」步骤 |
| 7 | **（可选）Ollama 服务** | 请求 `http://127.0.0.1:11434/api/tags` 或至少 `http://127.0.0.1:11434` 检测是否可访问 | 若用户可能选择「免费本地模型」，可提前知道 Ollama 是否已在运行、是否有模型列表；用于引导中步骤 5b 的初始状态 |
| 8 | **（可选）网络与权限** | 可选：对当前 npm registry 做一次 `GET` 或 `npm ping` 检测网络；可选：检查用户主目录、npm 全局 prefix 是否可写 | 用于提示「网络不可达请换源」或「权限不足请更换安装路径」 |

#### 检测完成后的逻辑

- **汇总结果**：得到一份结构化结果（如：Node 通过/未通过、npm 通过/未通过、OpenClaw CLI 通过/未通过、Daemon 通过/未通过、配置通过/未通过，以及各版本号、路径、registry）。
- **是否允许进入安装引导**：若运行环境（OS/架构）不支持，则**不进入**安装引导，仅展示提示与文档链接；其余情况允许进入。
- **引导起点**：根据检测结果，进入安装引导后**从「第一个未通过步骤」**开始展示（与 6.2 依赖链顺序一致）；若全部通过，可展示「全部就绪」或直接进入「配置/高级设置」。
- **与设置菜单联动**：若在功能检测前用户曾在设置中修改过 Node 路径或 npm 源，检测时必须使用这些配置（Node 路径用于执行 node/npm 命令，registry 仅记录用于后续安装）。

实现时可将上述操作封装为统一的「功能检测服务」，供「进入安装引导前」与「设置菜单中的重新检测环境」共用；检测过程可展示进度或简短说明（如「正在检测 Node…」「正在检测 OpenClaw…」），避免用户误以为卡死。

### 6.1 用户环境假设

- 用户**可能什么环境都没有**：无 Node、无 npm、无 OpenClaw，甚至不熟悉命令行。
- 应用需在**每一步**检测当前环境，并据此决定：跳过 / 引导安装 / 引导配置。
- 目标：无论用户从哪种状态进入，都能看到「当前缺什么、下一步该做什么」，并在一体化流程中完成。

### 6.2 依赖链与步骤拆解

安装与配置的**完整依赖链**（从前到后，任一步未满足则后续无法进行）：

| 步骤 | 依赖项 | 检测内容 | 未满足时的动作 |
|------|--------|----------|----------------|
| 0 | 操作系统与基础环境 | OS 类型、版本、架构（x64/arm64）、磁盘/内存概览 | 提示不支持的平台或资源不足，给出建议 |
| 1 | Node.js | 是否安装、版本是否 ≥ 24 | 引导安装 Node（官方安装包链接 / 包管理器 / 我们提供的下载与安装） |
| 2 | npm | 是否可用、版本 | 随 Node 通常已有；若缺失则单独引导 |
| 3 | OpenClaw CLI | 是否全局安装、版本 | 执行 `npm install -g openclaw@latest`，展示进度与结果 |
| 4 | OpenClaw Daemon/网关 | 是否已安装并运行 | **由本应用启动网关**（spawn `openclaw gateway`，推荐）或执行 `openclaw onboard --install-daemon`；检测 18789 端口是否就绪 |
| 5 | 配置文件与基础配置 | `~/.openclaw/openclaw.json` 是否存在、必要项是否填写（如 API Key 或**本地模型**） | 创建/编辑配置文件；用户可选「云端 API」或「免费本地模型」 |
| 5b | （可选）本地 Ollama | 若用户选「免费本地模型」：Ollama 是否运行、是否已选/已拉取模型 | 通过 electron-ollama 启动 Ollama，引导选择/拉取模型，再经 **Ollama 适配器** 写入 OpenClaw 配置 |

每一层都**先检测、再决策**：已满足则标记为完成并进入下一步；未满足则在本步提供明确操作（安装或配置），完成后再次检测，通过则进入下一步。步骤 5b 仅在用户选择「免费本地模型」时执行。

### 6.3 各步骤风险判断与处理方案

在每一步除「是否就绪」外，还需做**构建/运行风险判断**：识别环境被破坏或变更的情况，并给出处理方案与**图形化配置入口**，避免用户不知如何恢复。

| 步骤 | 典型风险 | 风险检测方式 | 处理方案 | 图形入口（配置/恢复） |
|------|----------|--------------|----------|------------------------|
| 0 | 系统升级后不兼容、磁盘满 | 定期再检测 OS/架构、磁盘空间 | 提示升级或清理，给出兼容性说明 | 设置 → 环境概览、诊断报告 |
| 1 | **Node 被卸载/更换版本/路径变更** | `node -v` 失败或版本 &lt; 24，或与上次记录的 Node 路径不一致 | 重新引导安装 Node；提示「检测到 Node 异常，请重新安装或选择正确路径」 | 设置 → **Node 路径/版本**、**重新检测环境** |
| 2 | npm 不可用（随 Node 重装一并丢失）；**npm 源在国内不可用或过慢** | `npm -v` 失败；可选：读取 `npm config get registry` 检测当前源 | 随步骤 1 恢复 Node 即恢复 npm；**提供更换 npm 源**（见下节） | 设置 → **npm 配置** → **源地址（registry）** |
| 3 | **Node 被删/换导致 npm 全局包全部丢失**（openclaw 等） | `openclaw -v` 或 which openclaw 失败 | **重新执行** `npm install -g openclaw@latest`（使用用户配置的 registry）；必要时先恢复/设置 npm 源 | 主流程「重新安装 OpenClaw」；设置 → **npm 源**、**重新安装全局依赖** |
| 4 | Daemon 未启动、崩溃、端口被占 | `openclaw doctor` 失败或网关不可达 | **由本应用启动网关**（推荐，兼容性更好）或重新执行 `onboard --install-daemon`；端口冲突时引导改端口或关闭占用进程 | 设置 → **由本应用启动网关** / 停止网关 / 安装并启动 Daemon |
| 5 | 配置文件被删、被损坏、被其他工具覆盖 | 读 `openclaw.json` 失败或必要字段缺失 | 从默认模板重建或合并；引导用户重新填写关键项 | 设置 → **打开 OpenClaw 配置**（编辑 `openclaw.json` 或表单） |
| 5b | Ollama 未运行、模型被删、端口冲突 | 11434 不可达；/api/tags 为空或模型不存在 | 重新启动 Ollama（electron-ollama 或本机）；重新选择或拉取模型；适配器重新写配置 | 设置 → **Ollama / 本地模型**（开关、模型列表、重新拉取） |

**设计原则**：

- **每次启动或「刷新」时**：对关键步骤做一次风险检测（Node/npm/openclaw/daemon/配置）；若某步从「已通过」变为「未通过」，在 UI 上标记为**异常/需恢复**，并突出「处理方案」按钮（如「重新安装 OpenClaw」「更改 npm 源」）。
- **图形入口统一**：在应用内提供「**设置**」或「**配置中心**」入口，集中展示上述可调项（Node、npm 源、OpenClaw 配置、Daemon、Ollama），用户可随时修改或触发「重新检测 / 重新安装」，而不必重新走完整向导。

### 6.4 每一步的环境监测（实现要点）

（与 6.2、6.3 配合：检测结果既用于「是否就绪」，也用于「风险判断」。）

- **步骤 0**
  - 使用 Node/Electron 的 `os`、`process` 等获取：`platform`、`arch`、内存与磁盘（可选）。
  - 不满足时：提示「当前系统/资源不满足」，并给出文档或下载链接，不执行后续安装。

- **步骤 1～2**
  - 在**子进程**中执行 `node -v`、`npm -v`（注意使用用户环境的 PATH，或约定好的安装路径）。
  - Windows 上若用户刚安装 Node，可能需提示「请重启应用以刷新 PATH」或提供「刷新环境」按钮再次检测。

- **步骤 3～4**
  - 执行 `openclaw -v` 或 `openclaw doctor` 判断 CLI 与 daemon 状态。
  - 安装/启动命令在应用内执行，并捕获 stdout/stderr，将进度与结果反馈到 UI（日志或进度条）。

- **步骤 5**
  - 读 `~/.openclaw/openclaw.json`（跨平台用 `os.homedir()`），判断是否存在及必要字段。
  - 缺失则创建默认结构；不完整则引导用户在应用内表单填写或跳转官方配置方式。若用户选择「免费本地模型」，则进入步骤 5b。

- **步骤 5b（本地模型）**
  - 检测 `http://127.0.0.1:11434` 是否可访问（electron-ollama 或本机 Ollama）；若未运行则通过 electron-ollama 启动。
  - 调用 Ollama API（如 `/api/tags`）获取已拉取模型列表，供用户选择；支持 `ollama pull <model>` 拉取新模型。
  - Ollama 适配器根据用户所选模型，向 `openclaw.json` 写入 OpenAI 兼容的 provider（`baseUrl: http://127.0.0.1:11434/v1`），并设置对应 model 条目。

### 6.5 根据当前环境继续完成功能搭建

- **状态持久化**：可本地记录「当前完成到第几步、各步检测结果」，便于下次启动时直接展示状态，而不是从头再测一遍（可选）。
- **入口统一**：主界面始终展示「整体进度」（例如：Node ✓、npm ✓、OpenClaw CLI ✓、Daemon 待启动、配置待完成），用户点击某一步可单独「检测」或「执行」该步。
- **断点续传**：用户可能中途关闭应用。再次打开时，重新跑一遍各步检测，从**第一个未完成步骤**继续引导，避免重复操作已完成的步骤。
- **降级与提示**：某步失败时（如网络错误、权限不足），明确提示原因与建议（如「请检查网络」「请以管理员运行」），并允许重试或跳过（跳过则标记为未完成，后续仍可再试）。

### 6.6 流程小结

1. **阶段一（安装）**：进入前先执行**功能检测**（6.0）；进入后按依赖链顺序展示并执行各步（0～5，可选 5b），**安装过程展示进度**，**询问是否免费模型**，右上角可随时进入设置。
2. **阶段二（OpenClaw 配置）**：安装就绪后，**图形化列出基础配置项**（见 1.2），用户选择/填写并写回 openclaw.json，点击「完成配置」后**退出安装向导**。
3. **阶段三（主界面）**：应用切换为 **Agent 对话**（上对话区、下输入框+发送按钮，发送旁设置按钮），供日常使用。
4. **交互**：阶段一内用户可执行当前未通过步骤或单独某步；每步执行后再次检测；设置与安装共用同一套检测与执行逻辑。

这样即可实现：**为所有用户创建友好的安装与配置流程，补全 OpenClaw 配置环节，并在每一步监测用户环境，根据当前已有环境继续完成功能搭建**。

### 6.7 配置项分析与可调项（含 npm 源地址）

应用会涉及多处配置；需明确**有哪些配置项**、**哪些支持用户修改**，并统一提供**图形入口**（尤其是中国用户常用的 **npm 源地址**）。

### 配置项一览

| 配置项 | 作用范围 | 存储位置/方式 | 是否支持用户在应用内修改 | 说明 |
|--------|----------|----------------|--------------------------|------|
| **npm 源地址（registry）** | 步骤 2、3：npm 安装全局包时的下载源 | 用户级 `~/.npmrc` 或 `npm config set registry` | **是**（必须支持） | 中国用户常需改为国内镜像，否则安装慢或失败 |
| Node 安装路径 / 可执行路径 | 步骤 1、2、3：检测与执行 node/npm 时的 PATH | 应用内可选「记忆」或用户指定 | 可选 | 多版本/自定义安装时有用 |
| OpenClaw 网关与模型配置 | 步骤 5：OpenClaw 运行时使用的模型、API、网关等 | `~/.openclaw/openclaw.json` | **是** | 表单或「打开配置文件」入口 |
| OpenClaw Daemon 开关/端口 | 步骤 4 | 由 OpenClaw 自身管理，可能写入 openclaw.json 或环境变量 | 是（通过配置或服务管理） | 启动/停止/重启；端口冲突时引导修改 |
| Ollama 模型选择与拉取 | 步骤 5b | 应用内选择 + 通过适配器写入 openclaw.json | **是** | 模型列表、默认模型、拉取新模型 |
| electron-ollama 数据目录 | 步骤 5b | `app.getPath('userData')` 等 | 可选（高级） | 一般用默认即可 |

### 重点：支持用户更换 npm 源地址

- **原因**：中国用户访问官方 npm registry 可能较慢或受限，需要更换为国内镜像（如淘宝镜像、npmmirror）才能稳定完成 `npm install -g openclaw` 等操作。
- **实现要点**：
  - **检测当前源**：在步骤 2/3 执行前或「设置」页，执行 `npm config get registry`（或读 `~/.npmrc`），展示当前 registry。
  - **提供可选项**：在应用内提供「**npm 源地址**」配置入口，至少支持：
    - **默认（官方）**：`https://registry.npmjs.org/`
    - **国内常用镜像**：如 `https://registry.npmmirror.com`（原淘宝镜像）、或其他可配置的预设；
    - **自定义**：用户填写任意 registry URL。
  - **写入方式**：通过子进程执行 `npm config set registry <url>`，或直接读写用户目录下的 `.npmrc`（注意作用域：全局 vs 项目，我们关心全局/用户级）。
  - **生效时机**：修改后，后续所有在本应用中触发的 `npm install -g openclaw` 等命令将自动使用新源；可提示用户「已更换源，下次安装将使用新地址」。
- **图形入口**：设置 → **npm 配置** → **源地址（registry）**：下拉或输入框 + 预设（官方 / npmmirror / 自定义）+ 保存与「重新检测」或「重新安装 OpenClaw」。

### 配置入口汇总（与「必备设置菜单」对应）

- **必备设置菜单**（见 1.1）中**至少**包含以下三项，其余为推荐或可选：
  - **Node 环境地址**：Node 可执行路径或安装目录，用于检测与执行命令；重新检测环境。
  - **npm 源地址（registry）**：预设（官方 / npmmirror 等）+ 自定义；可选代理。
  - **OpenClaw 包**：版本、重新安装全局包、打开/编辑配置文件、Daemon 服务管理（启停/重启）等。
- 可选：诊断日志、**本地模型（Ollama）**（开关、模型列表、默认模型、拉取新模型）。

这样即可在**每一步引入风险判断与处理**，并**明确配置项、支持用户更换 npm 源**，满足中国用户与「环境被破坏后恢复」的需求。

## 七、功能模块设想（与流程对应）

1. **环境检测与风险判断服务（对应步骤 0～5 / 5b，及 6.0 功能检测）**
   - **进入安装引导前的功能检测**：按 6.0 执行运行环境、Node、npm、OpenClaw CLI、Daemon、配置文件及可选 Ollama 等检测，产出汇总结果，供「是否进入引导、从哪一步开始」使用。
   - 封装各步检测逻辑（Node/npm/OpenClaw/daemon/配置文件、Ollama 是否运行），返回结构化结果（是否就绪、版本、路径、错误信息）。
   - **风险判断**：对比「上次状态」或预期状态，识别 Node 被删/换、全局包丢失、配置损坏等，返回风险类型与建议操作。
   - 支持「单步检测」与「全量检测」，供功能检测环节、安装引导主流程与设置菜单中的「重新检测」使用。

2. **分步安装与配置执行器**
   - 根据检测结果执行：Node 安装引导、`npm install -g openclaw`、`onboard --install-daemon`、配置文件创建/编辑。
   - 子进程管理、输出捕获、超时与错误处理，结果回传给渲染进程。

3. **electron-ollama 集成与本地模型选择（对应步骤 5b）**
   - 集成 **electron-ollama**：在主进程按需启动/检测 Ollama（`basePath: app.getPath('userData')`），若无则 `eo.serve(version)`。
   - 提供模型列表（Ollama `/api/tags`）、用户选择默认模型、支持「拉取新模型」（`ollama pull`），并在 UI 展示进度。

4. **Ollama 适配器**
   - 根据用户选择的 Ollama 模型，生成 OpenClaw 所需的 **OpenAI 兼容** provider 配置（`baseUrl: http://127.0.0.1:11434/v1`、`api: "openai"`、模型 id/name/contextWindow 等），并写入 `~/.openclaw/openclaw.json`。
   - 可选：检测本机已有 Ollama（如 11434 端口）时，直接复用并只做配置写入，不强制使用内嵌 electron-ollama。

5. **阶段一：安装阶段 UI**
   - 对应 **1.1 阶段一**：展示依赖链每一步的状态（图标 + 简短说明），突出「当前需要做什么」；**安装过程需展示安装进度**（npm 日志、Daemon 启动等）；若某步从通过变为未通过，标记**异常/需恢复**并突出「处理方案」按钮。
   - 在安装流程中**询问用户是否启用免费本地模型**；若选是，进入步骤 5b（electron-ollama 安装/启动、模型选择与拉取、Ollama 适配器写配置）。
   - **右上角常驻「设置」按钮**，随时可切换 npm 源、Node 路径等；提供「执行当前步骤」「重试」「打开文档」等按钮。

6. **阶段二：OpenClaw 配置阶段 UI**
   - 对应 **1.1 阶段二** 与 **1.2 OpenClaw 基础配置项**：**图形化列出** OpenClaw 基础配置项（模型来源、API Key/端点、默认模型、本地模型、网关端口等），表单/卡片供用户选择或填写；提交后合并写回 `openclaw.json`。
   - 提供「完成配置」「退出安装向导」按钮，点击后进入**阶段三：主界面（Agent 对话）**。

7. **阶段三：主界面（Agent 对话）UI**
   - 对应 **1.1 阶段三**：**上方**为 AI 对话内容展示区，**下方**为输入框 + 发送按钮；**发送按钮旁设「设置」按钮**，进入设置菜单（与阶段一设置内容一致）。

8. **必备设置菜单（设置 / 配置中心）**
   - **入口**：阶段一安装界面**右上角**；阶段三主界面**输入框旁（发送按钮旁）**。
   - 须包含：**Node 环境地址**、**npm 源地址**、**OpenClaw 包**（版本、重新安装、打开配置目录、Daemon 服务管理）；可选 **本地模型（Ollama）**、诊断日志等。

9. **状态与帮助**
   - 可选：本地保存上次检测结果，下次启动先展示再后台刷新；用于风险对比（某步是否从通过变为未通过）。
   - 提供「打开官方文档」「检查更新」「诊断日志」等入口，便于排查问题。

## 十、llv-ollama 算力分布（规划摘要）

为突破**单机单 Ollama 端点**的并发与资源上限，规划引入 **llv-ollama**（工作名）：在 `openclaw.json` 的 provider 与多个 Ollama（`11434`、额外端口或局域网节点）之间增加**一层 HTTP 调度**，对外仍表现为单一 `baseUrl`（OpenAI 兼容），对内按策略将 `/v1/chat/completions` 分发到健康下游。

- **第一期**：独立小服务或主进程子模块 + 双节点 + 健康检查 + 流式透传。  
- **配置**：建议 `~/.openclaw/llv-ollama.json` 描述监听地址、下游列表与策略；OpenClaw 侧使用 `ollama-lv` 等 provider 指向 llv。  
- **详细设计、分期、风险**：见 **`doc/03-llv-ollama-算力分布方案.md`**。
