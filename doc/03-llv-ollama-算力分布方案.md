# llv-ollama：本地算力分布方案（设计稿）

> **命名**：`llv-ollama`（**L**oad-**L**ayer for **V**irtual Ollama pool，工作名）——在 OpenClaw 与多个 Ollama 实例之间增加一层**路由与观测**，实现「算力分布」（多实例、多机、多 GPU 下的请求调度与容错）。  
> **状态**：设计阶段；实现前需与产品确认命名（是否对外称「算力调度」「Ollama 集群」等）。

本文档与 `00-项目分析与规划.md`、`01-执行方案.md` 对齐，作为后续开发与联调依据。

---

## 一、背景与问题

### 1.1 现状（本仓库）

- **单点 Ollama**：通过 `electron-ollama` 或用户本机已安装的 Ollama，`openclaw.json` 中 `models.providers["ollama-local"]` 通常指向单一 `baseUrl`（如 `http://127.0.0.1:11434/v1`）。
- **OpenClaw 网关**仍是一条推理链：网关 →（配置中的 provider）→ Ollama。单机单实例时，**并发、多卡、局域网闲置机器**无法被利用。
- 已有 `doc/ollama-加速说明.md` 覆盖单机侧参数（`OLLAMA_NUM_PARALLEL`、量化、GPU 等），**不解决多实例路由**。

### 1.2 目标

| 目标 | 说明 |
|------|------|
| **算力分布** | 将推理请求分散到多个 Ollama 兼容端点（同机多端口 / 多机 / 多 GPU 各跑一实例）。 |
| **对 OpenClaw 透明** | 仍使用 **OpenAI 兼容** `POST /v1/chat/completions`（及可选 `GET /v1/models`），仅把 provider 的 `baseUrl` 从直连 `11434` 改为 **llv 监听地址**。 |
| **可运维** | 健康检查、熔断、基础指标（QPS、延迟、错误率）；便于在设置页展示「哪几个节点可用」。 |
| **渐进落地** | 先 **单机多实例**，再 **局域网多机**；不要求第一期就做 K8s 级编排。 |

### 1.3 非目标（第一期不做）

- 替代 OpenClaw 网关本身。
- 自动跨节点 **模型权重分片**（与 Ollama 单机多 GPU 的 `OLLAMA_SCHED_SPREAD` 等正交，可文档互链）。
- 强一致会话粘性（除非策略显式开启「同会话同节点」）。

## 二、总体架构

```
┌─────────────────────────────────────────────────────────────┐
│  OpenClaw 网关（既有）                                         │
│  读取 openclaw.json → provider「ollama-lv」或沿用 ollama-local │
└────────────────────────────┬────────────────────────────────┘
                             │ HTTP OpenAI 兼容
                             ▼
┌─────────────────────────────────────────────────────────────┐
│  llv-ollama（新增：调度层）                                    │
│  · 监听单一 baseUrl（如 http://127.0.0.1:11435/v1）            │
│  · 路由策略：轮询 / 最少连接 / 按模型亲和 / 加权               │
│  · 健康检查：/api/tags 或轻量 chat ping                       │
│  · 可选：简单管理 API（增删节点、权重）                        │
└────────────┬───────────────┬───────────────┬────────────────┘
             │               │               │
     ┌───────▼──────┐ ┌──────▼──────┐ ┌──────▼──────┐
     │ Ollama :11434 │ │ Ollama :11436│ │ 局域网 :11434│
     │ (GPU0)       │ │ (GPU1)       │ │ (同事机器)   │
     └──────────────┘ └─────────────┘ └─────────────┘
```

**关键点**：

- llv 在协议层表现为 **一个** OpenAI 兼容上游；下游可挂 **N 个** Ollama（或任意兼容服务）。
- 默认模型 ID 需在下游一致或做 **别名映射**（见第五节）。

## 三、组件形态与代码归属（建议）

| 方案 | 说明 | 适用 |
|------|------|------|
| **A. 独立 Node 服务（推荐 MVP）** | 小进程：`llv-ollama` CLI 或 `node dist/llv-server.js`，配置文件 `~/.openclaw/llv-ollama.json` | 易调试、与 Electron 解耦、可被非 GUI 使用 |
| **B. Electron 主进程内嵌** | 与现有 `ollama.ts` 同进程起 HTTP server | 少一个进程，但升级与端口生命周期与 App 强绑定 |
| **C. Rust/Go 独立二进制** | 高性能、单文件分发 | 后期若瓶颈在代理层再考虑 |

**建议**：**MVP 采用 A**，在「设置」中提供「安装/启动 llv-ollama 服务」或文档引导用户手动 `npx`/全局安装；与 **electron-ollama** 并行存在：electron-ollama 负责**拉起本机 Ollama**，llv 负责**多实例聚合**。

## 四、路由与策略

### 4.1 健康检查

- 周期：如每 **10～30s** 探测每个下游 `GET /api/tags` 或 `HEAD` 根路径（与现有 Ollama 版本兼容）。
- 失败：连续 **N 次** 失败则标记 `down`，不参与调度；恢复后自动 `up`。

### 4.2 调度策略（可配置）

| 策略 | 行为 | 适用场景 |
|------|------|----------|
| `round_robin` | 轮询可用节点 | 各节点算力相近 |
| `least_inflight` | 当前在飞请求最少 | 负载不均时 |
| `sticky_by_model` | 同一模型尽量打同一节点（利于 KV/缓存） | 多模型、少切换 |
| `weighted` | 按权重（如 GPU 显存比例） | 异构集群 |

第一期可实现 **round_robin + least_inflight** 二选一即可。

### 4.3 流式响应

- `stream: true` 时，llv 需 **绑定到选定下游连接**，中途不得换节点；连接失败时返回明确错误（不重试破坏 SSE 语义）。

## 五、配置模型（草案）

### 5.1 llv 侧配置（示例）

路径建议：`~/.openclaw/llv-ollama.json`（与 `openclaw.json` 同目录，便于备份）。

```json
{
  "listen": { "host": "127.0.0.1", "port": 11435 },
  "strategy": "least_inflight",
  "targets": [
    { "id": "local-a", "baseUrl": "http://127.0.0.1:11434", "weight": 1 },
    { "id": "local-b", "baseUrl": "http://127.0.0.1:11436", "weight": 1 }
  ],
  "healthCheck": { "intervalSec": 15, "timeoutMs": 3000, "failThreshold": 3 }
}
```

### 5.2 OpenClaw 侧

- 新增或复用 provider key，例如 **`ollama-lv`**，`baseUrl` 指向 `http://127.0.0.1:11435/v1`。
- 「一键应用本地模型」扩展：若启用 llv，写入 **ollama-lv** 的 `baseUrl` 与模型列表（从 llv 聚合 `GET /v1/models` 或合并各节点）。

### 5.3 模型 ID 与列表

- **聚合 `/v1/models`**：合并各节点返回的 `id`，去重；冲突时加后缀或取并集策略需在实现时写死一条规则。
- **默认模型**：用户选的 id 必须至少在一个存活节点存在，否则在 UI 标红。

## 八、风险与依赖

- **Ollama 版本差异**：不同节点模型列表不一致时，需在 UI 明确提示。
- **安全**：局域网暴露 Ollama 需 TLS/防火墙策略（文档说明，默认仅 127.0.0.1）。
- **与 `ollama-加速说明.md` 关系**：单机优化仍有效；llv 解决**横向**扩展，不替代单实例参数调优。
