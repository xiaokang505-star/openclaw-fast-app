## 十、相对完善版 GUI 操作路径（新增规划）

### 10.1 目标与边界

- 目标：让 OpenClaw 在本应用内对「打开应用、窗口切换、点击输入、文件选择、截图确认」等 GUI 任务具备**可执行、可观测、可回滚**能力，而不是只返回文字建议。
- 边界：仅覆盖用户授权的本机 GUI 操作；不默认开启高风险动作（如批量删除、系统设置改写、自动付款）。

### 10.2 端到端路径（建议实现）

1. **意图识别层（Renderer/Main）**
   - 在发送前做 GUI intent 识别：如「打开微信」「点击保存」「在设置里开启蓝牙」。
   - 命中 GUI intent 时，强制使用 `chatBackend=openclaw`，不走 `ollama` 直连。
2. **策略编排层（Main）**
   - 注入 GUI 模式系统约束：优先本机工具、禁止跨渠道 message/sessions_send、失败必须回传可执行下一步。
   - 按风险等级决策：低风险自动执行；中风险二次确认；高风险阻断并给替代方案。
3. **执行适配层（Gateway/Tool Adapter）**
   - 统一封装 GUI 动作：`open_app`、`focus_window`、`click`、`type`、`hotkey`、`screenshot`、`assert_ui`。
   - 所有动作都写审计日志（动作、参数、耗时、结果、错误码、截图快照路径）。
4. **结果闭环层（Main/Renderer）**
   - 每次动作返回结构化结果并回写到会话：`success | failed | blocked` + reason。
   - 对失败自动进入诊断分支：权限不足、应用未安装、元素未找到、窗口被遮挡等。
5. **恢复与重试层**
   - 失败后自动尝试有限重试（指数退避，最多 N 次），并支持用户「继续重试 / 改为手动」。

### 10.3 配置与控制面（OpenClaw 配置页新增）

- **GUI 总开关**
  - `gui.enabled`（默认 false，首次启用需用户确认）。
- **能力白名单**
  - `gui.allowApps`（仅允许操作的应用，如 Finder/WeChat/Chrome）。
  - `gui.allowActions`（允许动作子集，如 open/click/type，默认不含 destructive）。
- **风险控制**
  - `gui.requireConfirmFor`（高风险动作列表）。
  - `gui.maxStepPerTask`（单任务最大 GUI 动作数，防止失控循环）。
- **执行体验**
  - `gui.captureScreenshotOnStep`（每步截图/关键步截图）。
  - `gui.stepTimeoutMs`、`gui.retryPolicy`。

### 10.4 安全模型（必须项）

- **最小权限**：仅在用户显式启用 GUI 后开放；未启用时统一返回「能力未开启」。
- **双确认机制**：高风险动作执行前弹确认；支持「本次允许」「总是允许此应用/动作」。
- **可审计性**：每条 GUI 动作保留日志与截图链路，便于追责与回放。
- **硬阻断规则**：涉及支付、账号安全、隐私导出时默认阻断，除非用户临时授权。

### 10.5 错误分类与文案规范

- `GUI_PERMISSION_DENIED`：缺少 macOS 辅助功能/屏幕录制权限。
- `GUI_APP_NOT_FOUND`：目标应用不存在或名称不匹配。
- `GUI_ELEMENT_NOT_FOUND`：控件未出现或定位失败。
- `GUI_TIMEOUT`：页面响应慢/动画未完成。
- `GUI_POLICY_BLOCKED`：被白名单/风险策略拦截。

> 统一返回格式：`code + userMessage + debugHint + nextAction`，避免仅显示「执行失败」。

### 10.6 观测与验收指标

- GUI 任务成功率（按任务类型分层）。
- 单步平均耗时、P95。
- 自动重试后成功率提升比例。
- 人工介入率（需要用户改手动的占比）。
- 高风险误执行率（目标为 0）。

### 10.7 分阶段落地计划（非最小版）

1. **Phase A：控制与审计先行**
   - 先完成配置项、安全策略、日志链路、统一错误码。
2. **Phase B：核心 GUI 动作闭环**
   - 实现 open/focus/type/click/screenshot/assert 六类动作与重试策略。
3. **Phase C：任务级编排**
   - 支持多步 GUI 任务计划、步骤回放、失败分支自动恢复。
4. **Phase D：稳定性与体验优化**
   - 模型提示词优化、元素定位鲁棒性、跨分辨率适配、异常回归测试集。

### 10.8 测试计划（建议）

- **功能用例**：打开微信、打开 Finder、切换窗口、保存文件、系统偏好设置导航。
- **异常用例**：应用未安装、权限未授权、窗口遮挡、分辨率变化、多显示器。
- **安全用例**：高风险动作确认、白名单阻断、日志完整性验证。

| 步骤 1 | Node.js | Node 已安装且版本 ≥ 24 |
| 步骤 2 | npm | npm 可用、registry 可配置 |

## 二、功能检测环节（进入安装引导前）

### 2.1 执行方案

- **触发**：用户点击「开始安装」或首次进入安装引导时执行；设置菜单中「重新检测环境」可再次触发。
- **操作**：按 00 文档 6.0 顺序执行：运行环境 → Node → npm → OpenClaw CLI → Daemon → 配置文件 →（可选）Ollama →（可选）网络/权限。
- **实现**：主进程内封装「功能检测服务」；子进程调用 `node -v`、`npm -v`、`npm config get registry`、`openclaw -v`、`openclaw doctor` 等；读 `path.join(os.homedir(), '.openclaw', 'openclaw.json')`；HTTP 探测 18789、11434 端口；汇总为结构化结果（各步通过/未通过、版本、路径、registry）。
- **产出**：检测报告 + 是否允许进入安装引导 + 引导起点（第一个未通过步骤）。

### 2.2 痛点

- 子进程执行依赖「当前 PATH」或「设置中保存的 Node 路径」：若用户刚安装 Node 未重启应用，Windows 上可能拿不到新 PATH，检测结果失真。
- 检测项多、串行执行时耗可能较长，若 UI 无进度反馈易被误认为卡死。
- 部分环境（如企业代理、防火墙）下 HTTP 探测 18789/11434 可能超时或误判。

### 2.3 风险点

- **误判「不支持」**：未正确识别 platform/arch（如 Electron 在 arm64 Mac 上 process.arch 为 arm64），导致不进入引导或下载错误架构的包。
- **检测顺序依赖**：若先测 OpenClaw 再测 Node，Node 未安装时子进程报错需妥善捕获，避免整轮检测失败无结果。
- **跨平台路径**：Windows 下 homedir 与路径拼接错误（如混用 `/` 与 `\`）导致读不到 openclaw.json，误判为「未配置」。

### 2.4 注意点

- 使用 `process.platform`、`process.arch` 与 `os.homedir()`、`path.join()` 保证跨平台；子进程环境需传入或继承正确的 PATH 或指定 Node 可执行路径。
- 检测过程提供简要进度文案（如「正在检测 Node…」「正在检测 OpenClaw…」）或进度条，超时设置合理（如单步 10–15 秒）。
- 任一子步骤失败时记录错误信息并继续后续步骤，最终报告中标明「未通过」及原因，便于用户或设置菜单中重试/修正。

## 三、步骤 0：运行环境

### 3.1 执行方案

- **检测**：读取 `process.platform`、`process.arch`；可选 `os.totalmem()`/`os.freemem()`、磁盘剩余空间（需 Node 或系统 API）。
- **判断**：若 (platform, arch) 不在支持列表（如 win32+ x64/arm64，darwin+ x64/arm64，linux+ x64/arm64）则标记不支持；可选：内存或磁盘低于阈值时仅提示仍允许继续。
- **未通过时**：不进入安装引导，展示「当前系统/架构不支持」及文档链接；或「资源不足建议」仍允许用户选择继续。

### 3.2 痛点

- 磁盘空间检测在 Node 标准库中无直接 API，需调用系统命令或第三方包，增加依赖与跨平台差异。
- 用户可能使用 WSL2：若在 Windows 本机运行 Electron，检测到的是 win32，无法直接反映 WSL 内环境；若需支持「在 WSL 内安装 OpenClaw」需单独说明或另做检测逻辑。

### 3.3 风险点

- **架构与安装包不一致**：检测为 arm64 但后续下载/安装脚本误用 x64 的 Node 或 Ollama 包，导致无法运行。
- **过严限制**：将未明确测试的 OS/arch 直接判为不支持，导致部分用户无法使用；建议明确支持矩阵并在文档中说明「实验性」支持。

### 3.4 注意点

- 支持矩阵与 00 文档 3.A、3.C 一致；检测结果中的 arch 必须传递到「Node 安装引导」「electron-ollama」等后续步骤，用于选择对应架构的包或二进制。
- 内存/磁盘为可选增强，不阻塞主流程；若实现，阈值需文档化（如最低 8GB 内存、10GB 磁盘）。

## 四、步骤 1：Node.js

### 4.1 执行方案

- **检测**：使用设置中的 Node 路径（若有）或系统 PATH，子进程执行 `node -v`，解析版本号（需 ≥ 24）；可选 `node -p process.execPath` 得到可执行路径并保存供后续使用。
- **未通过时**：引导用户安装 Node——提供官方下载链接（按 platform+arch 拼接，如 nodejs.org 或镜像）、或包管理器命令（如 winget、brew）；若有「应用内下载并安装」则需实现解压/安装到用户目录并写入 PATH 或记录路径到设置。
- **通过后**：将检测到的 Node 路径（若有）写入或更新到设置，供步骤 2、3 使用。

### 4.2 痛点

- **Windows PATH**：用户通过官方安装包安装 Node 后，新开的终端才有新 PATH；Electron 进程若未重启，子进程继承的仍是旧环境，检测不到新装的 Node，需提示「请重启本应用」或提供「刷新环境」并重启/重新 spawn。
- **多版本共存**：用户已装 nvm/fnm 等，系统 PATH 中可能指向某版本，与「默认 node」不一致；需支持在设置中指定「Node 可执行路径」并在检测与后续安装中统一使用该路径。
- **企业环境**：无外网、需代理或自定义安装路径时，官方链接可能无法访问，需依赖「设置中的 Node 路径」或手动安装后由用户指定。

### 4.3 风险点

- **版本解析错误**：`node -v` 输出格式因版本/语言可能不同（如 `v22.0.0` 或带换行），解析失败时勿误判为「已安装但版本未知」，应明确区分「未安装」与「版本无法识别」。
- **权限**：安装到系统目录（如 ProgramFiles）需管理员权限，安装失败时需提示「请以管理员运行」或改为用户目录安装并配置 prefix。
- **下载/安装中断**：若实现应用内下载 Node 安装包，需处理网络中断、磁盘满、杀毒软件拦截等，并提供重试与明确错误提示。

### 4.4 注意点

- 所有后续依赖 Node/npm 的子进程，统一使用「设置中保存的 Node 路径」或检测到的 execPath 作为可执行文件所在目录，并确保该目录的 `node`、`npm`（及 Windows 下 `npm.cmd`）可被 spawn。
- 官方 Node 下载 URL 需按 (platform, arch) 动态拼接；arm64 必须使用 arm64 包，不可用 x64 替代。
- 安装完成后在 UI 明确提示 Windows 用户「若检测仍显示未安装，请关闭并重新打开本应用」。

## 五、步骤 2：npm

### 5.1 执行方案

- **检测**：在与步骤 1 相同的环境中执行 `npm -v`、`npm config get registry`；确认 npm 可用且版本可接受（通常随 Node 安装即可）。
- **未通过时**：通常与 Node 未安装或路径错误同时出现，先解决步骤 1；若 Node 已安装但 npm 不可用（极少见），提示重装 Node 或检查 PATH。
- **配置**：在设置菜单中提供「npm 源地址」：读取当前 registry，提供预设（官方、npmmirror 等）与自定义 URL；保存时执行 `npm config set registry <url>` 或写用户级 `.npmrc`，确保后续 `npm install -g openclaw` 使用该源。

### 5.2 痛点

- **中国用户**：默认 registry 在国外，安装 openclaw 时可能极慢或超时，必须在步骤 3 前或设置中提供换源入口，否则步骤 3 失败率高。
- **作用域**：`npm config set registry` 可能写全局或用户级；需明确使用用户级（如 `--location=user` 或写 `~/.npmrc`），避免影响系统其他项目。
- **企业代理**：部分环境需配置 proxy；若仅改 registry 仍无法访问，需在文档或设置中说明代理配置方式（如 npm config set proxy）。

### 5.3 风险点

- **registry 未生效**：子进程环境未继承用户级 .npmrc 或 npm config，导致安装时仍用默认源；需在 spawn 时指定 env 或确保工作目录/主目录正确。
- **自定义 URL 格式错误**：用户输入缺少协议或尾斜杠导致 npm 报错；保存前做基本校验（如以 https?:// 开头、可选的 / 结尾）。
- **镜像与官方不同步**：国内镜像延迟或某些包缺失，安装 openclaw 时可能报 404 或版本不对；建议提示「若安装失败可尝试切换为官方源或其它镜像」。

### 5.4 注意点

- 步骤 3 执行 `npm install -g openclaw@latest` 前，必须使用设置中已保存的 registry（通过 env 或先执行一次 `npm config set registry`）；功能检测与安装引导共用同一配置来源。
- 写 `.npmrc` 时路径必须用 `path.join(os.homedir(), '.npmrc')`，Windows 下 homedir 正确。
- 可选：在功能检测或设置页对当前 registry 做一次简单可达性检测（如 GET 或 npm ping），失败时提示「当前源可能不可达，建议更换」。

## 六、步骤 3：OpenClaw CLI

### 6.1 执行方案

- **检测**：在含 Node 全局 bin 的 PATH 下（或使用设置中的 Node 路径所在目录及其同级的 npm 全局 prefix）执行 `openclaw -v` 或 `openclaw --version`；能解析出版本即通过。
- **未通过时**：执行 `npm install -g openclaw@latest`（或设置中指定的版本），使用已配置的 registry；子进程 stdout/stderr 实时输出到 UI（日志或进度条）；安装完成后再次检测。
- **通过后**：可记录 openclaw 版本，供步骤 4、5 使用。

### 6.2 痛点

- **安装耗时**：npm 全局安装可能较久，需展示进度或「正在安装…」避免用户以为卡死；可考虑解析 npm 输出中的进度信息若可用。
- **网络/registry**：若 registry 不可达或 openclaw 包很大，易超时或失败；需明确错误提示（如「网络超时，请检查 npm 源或网络」）并允许重试或跳转到设置更换源。
- **全局 prefix 与 PATH**：Windows 下 npm -g 默认装到 `%AppData%\npm`，该目录需在 PATH 中才能直接执行 `openclaw`；若用户自定义了 prefix，spawn 时需使用包含该 prefix/bin 的 PATH。

### 6.3 风险点

- **权限**：写入 npm 全局目录失败（如无写权限）时，npm 会报错；需捕获并提示「请以管理员运行」或「请修改 npm 全局 prefix 到用户目录」。
- **版本冲突或损坏**：已有 openclaw 但版本过旧或损坏，`openclaw -v` 可能报错；可建议「重新安装」并执行 `npm install -g openclaw@latest --force` 或先 uninstall 再 install。
- **杀毒/安全软件**：部分环境会拦截 npm 或 node 写可执行文件，导致安装「成功」但 openclaw 命令不可用；除提示重试外，可建议用户临时放行或添加排除目录。

### 6.4 注意点

- 执行 npm 的子进程需继承或显式设置 env：PATH 包含 Node 与 npm 全局 bin，若使用了用户级 .npmrc 需确保 NPM_CONFIG_USERCONFIG 或 HOME 正确，以便 npm 读到 registry 配置。
- 安装命令建议写为可配置（如从设置读取「OpenClaw 包版本」），默认 `openclaw@latest`。
- 安装失败时保留 stderr 全文用于诊断，并在 UI 提供「复制错误信息」「打开设置更换 npm 源」「重试」等入口。

## 七、步骤 4：OpenClaw Daemon

### 7.1 执行方案

- **检测**：执行 `openclaw doctor` 或 HTTP 请求网关端口（如 `http://127.0.0.1:18789`）判断 Daemon 是否已安装并运行。
- **未通过时**：执行 `openclaw onboard --install-daemon`（或 OpenClaw 官方文档中推荐的等价命令）；该过程可能为交互式，需评估是否需传非交互参数或引导用户在外层终端执行；若为后台服务，安装后需检测服务是否在运行。
- **通过后**：可展示「网关已就绪」及控制台 URL（如 18789），供步骤 5 配置或用户自行打开。

### 7.2 痛点

- **交互式 onboard**：`onboard --install-daemon` 可能需用户输入或确认，在 Electron 子进程内直接执行可能挂起；需查阅 OpenClaw 文档是否支持无交互参数，或改为「在新终端中打开并执行」由用户完成。
- **端口占用**：若 18789 已被占用，Daemon 可能无法启动或绑定失败；需能检测并提示「端口被占用，请关闭占用程序或修改 OpenClaw 配置中的端口」。
- **开机自启**：用户可能期望 Daemon 开机自启，需区分「本次已启动」与「已配置为系统服务/自启」，后者可能需额外配置或文档说明。

### 7.3 风险点

- **安装/启动超时**：Daemon 首次安装或启动可能较慢，子进程或 HTTP 探测需合理超时并允许重试。
- **权限与防火墙**：Windows 防火墙或安全软件可能拦截本地监听端口，导致本机请求 127.0.0.1 也失败；需在文档或提示中说明「若检测失败请检查防火墙/安全软件」。
- **版本兼容**：CLI 与 Daemon 版本不匹配时可能出现异常；尽量使用同一来源的 openclaw 版本（如均通过 npm -g 安装的 latest）。

### 7.4 注意点

- 网关端口以 OpenClaw 官方文档为准（如 18789），若可配置则需在读取 openclaw.json 或环境变量后使用实际端口做探测。
- 若 onboard 必须交互，可提供「在终端中打开」按钮，并给出需执行的命令文案，由用户在本机终端完成后再回到应用点击「重新检测」。

## 八、步骤 5：配置文件与基础配置

### 8.1 执行方案

- **检测**：读 `path.join(os.homedir(), '.openclaw', 'openclaw.json')`，检查文件是否存在、可读、且为合法 JSON/JSON5；检查是否包含必要字段（如 models、gateway 等，以 OpenClaw 文档为准）。
- **未通过时**：若文件不存在则创建目录与默认 openclaw.json 模板；若存在但不完整则引导用户在应用内表单填写（如 API Key、模型来源）或跳转「打开 OpenClaw 控制台」「运行 openclaw configure」；若用户选择「免费本地模型」则进入步骤 5b。
- **写入时**：任何应用内修改配置都需注意 JSON5/JSON 格式、备份（可选）、写回后提示「配置已更新，部分变更可能需要重启 OpenClaw 或 Daemon」。

### 8.2 痛点

- **配置结构复杂**：openclaw.json 可能包含大量字段，应用内表单难以覆盖全部；建议只做「最小必要项」编辑（如默认模型、API Key、网关地址），其余提供「在编辑器中打开」或「打开控制台」入口。
- **JSON5 解析**：若 OpenClaw 使用 JSON5，需使用支持 JSON5 的解析库，否则带注释的配置会解析失败。
- **并发写**：用户可能在应用写配置的同时用其他工具或控制台修改，存在覆盖风险；若可实现，写前再次读并合并，或提示「请勿在外部同时编辑」。

### 8.3 风险点

- **路径错误**：Windows 下 `os.homedir()` 与 `.openclaw` 拼接错误（如反斜杠、编码）导致读错文件或写到错误位置；必须用 path.join 并统一使用 UTF-8。
- **必要字段理解偏差**：OpenClaw 升级后必要字段可能变化，应用内校验过严会导致误判「未配置」；可依赖 `openclaw doctor` 或官方文档做兼容。
- **敏感信息**：API Key 等写入配置文件后，在「打开配置」「诊断日志」等入口需避免明文泄露（如打码或提示用户自行保管）。

### 8.4 注意点

- 支持 `OPENCLAW_STATE_DIR` 或 `OPENCLAW_HOME` 等环境变量时，配置路径可能不在默认 homedir；若需兼容，可在检测时先查环境变量再回退到 `~/.openclaw`。
- 创建默认配置时参考 OpenClaw 官方示例，避免缺必填项导致 Daemon 或 CLI 报错。
- 步骤 5 与步骤 5b（Ollama）的配置写入需合并到同一 openclaw.json，避免互相覆盖。

## 九、步骤 5b：本地 Ollama（可选）

### 9.1 执行方案

- **触发**：仅在用户选择「免费本地模型」时执行。
- **检测**：请求 `http://127.0.0.1:11434` 或 `/api/tags` 判断 Ollama 是否运行；若未运行则通过 electron-ollama 在应用内启动（使用 `app.getPath('userData')` 等），或提示用户安装/启动本机 Ollama。
- **模型**：拉取 `/api/tags` 得到已安装模型列表，供用户选择默认模型；支持「拉取新模型」即子进程执行 `ollama pull <model>`，并展示进度。
- **适配器**：根据用户所选模型，生成 OpenClaw 的 OpenAI 兼容 provider 配置（baseUrl: `http://127.0.0.1:11434/v1`，api: openai，models 列表），合并写入 `~/.openclaw/openclaw.json`；写完后提示可运行 `openclaw doctor` 或重启 Daemon 使配置生效。

### 9.2 痛点

- **electron-ollama 体积与下载**：内嵌 Ollama 需下载对应平台与架构的二进制，首次启动可能较慢；需展示下载/解压进度并处理网络失败。
- **与本机 Ollama 冲突**：用户本机已装 Ollama 且占用 11434 时，electron-ollama 不应再启动一套；需先检测 11434 是否已可用，若已可用则直接复用并只做配置写入。
- **模型拉取耗时**：ollama pull 大模型时可能很久，需后台或子进程 + 进度输出，避免 UI 假死。

### 9.3 风险点

- **架构与二进制**：electron-ollama 提供的二进制需与当前 (platform, arch) 一致，否则无法运行；与步骤 0 的 arch 检测结果一致使用。
- **端口冲突**：若 11434 被其他进程占用，Ollama 无法启动；需检测并提示「端口 11434 已被占用」。
- **配置覆盖**：Ollama 适配器写入 openclaw.json 时若直接覆盖整个 models 节点，可能抹掉用户已有的其它 provider；应以「合并/追加」方式写入 ollama 的 provider，保留其它配置。

### 9.4 注意点

- 模型 id、contextWindow、maxTokens 等需与 Ollama 实际能力或文档一致，避免 OpenClaw 调用时报错。
- 若用户切换默认模型，适配器需更新 openclaw.json 中对应 provider 的 models 列表或默认模型引用。
- electron-ollama 的 basePath 与版本号（如 v0.11.0）需在依赖与文档中固定，避免自动升级导致不兼容。

## 十、阶段二：OpenClaw 配置流程

（补全「安装完成后、进入主界面前」的**配置漏洞**：图形化完成 OpenClaw 基础配置，再退出向导。）

### 10.1 执行方案

- **触发**：阶段一（步骤 0～5 / 5b）全部就绪后，进入**阶段二**；或检测到 openclaw.json 已存在但必要项未填时，引导用户进入本阶段。
- **界面**：以表单或卡片**图形化列出** OpenClaw 基础配置项（与 00 文档 1.2 对应）：
  - **模型来源**：云端 API / 本地 Ollama（若已装）；单选。
  - **API Key / 端点**（若选云端）：输入框；可选从环境变量读取。
  - **默认模型**：下拉或输入（从当前 provider 的 models 拉取或手动填）。
  - **本地模型**（若选本地）：展示已选 Ollama 模型、可切换或拉取新模型（若 5b 已做可只做展示/切换）。
  - **网关端口**（可选）：数字输入，默认如 18789。
  - 其它必要项按 OpenClaw 文档；或提供「高级：打开配置文件」。
- **数据**：若已有 `~/.openclaw/openclaw.json`，读取并**预填**表单；用户提交后**合并写回**（不覆盖未在表单中编辑的键），注意 JSON5 格式与备份（可选）。
- **完成**：用户点击「完成配置」或「退出安装向导」后，**安装向导结束**，应用切换为**阶段三：主界面（Agent 对话）**。

### 10.2 痛点

- openclaw.json 结构可能随 OpenClaw 版本变化，表单字段需与文档同步或做兼容。
- 用户可能已在外部编辑过配置，合并写回时需避免覆盖用户自定义键；敏感信息（API Key）需安全展示与存储。

### 10.3 风险点

- **写回失败**：路径错误、权限不足、JSON 格式错误导致写回失败；需捕获错误并提示，保留「打开配置目录」由用户手动编辑。
- **与 5b 重复**：若用户已在 5b 选了 Ollama 模型并写入了 provider，阶段二表单应展示为已选状态，避免重复选择或覆盖。

### 10.4 注意点

- 阶段二与步骤 5「配置文件与基础配置」区分：步骤 5 解决「文件是否存在、是否可创建」；阶段二解决「**基础项内容**由用户图形化填写并写回」。
- 实现时与 00 文档 1.2「OpenClaw 基础配置项」表对齐，后续可随 OpenClaw 文档扩展字段。

## 十一、阶段三：主界面（Agent 对话）

（安装与配置全部完成后，应用主界面呈现 Agent 对话样式。）

### 11.1 执行方案

- **触发**：用户完成阶段二并点击「完成配置」/「退出安装向导」后，应用界面**切换**为主界面；或用户再次打开应用且已完成过向导时，直接进入主界面。
- **对话后端**：固定使用 **OpenClaw 网关**（POST `/v1/chat/completions`），不提供 Ollama/OpenClaw 切换；需 Daemon 已启动且配置中 `gateway.http.endpoints.chatCompletions.enabled: true`。
- **布局**：
  - **上方**：可滚动的 AI 对话内容区（消息列表、角色区分、流式输出；**回复支持 Markdown 渲染**，marked + DOMPurify）。
  - **底部固定**：**输入框** + **发送按钮** + **「设置」按钮**；输入框一行不随内容多少移动（flex 布局，消息区 overflow auto）。
- **与 OpenClaw 联动**：主进程 `openclaw-chat.ts` 读配置 port/token，请求网关；若返回 **HTTP 404**，错误文案会提示用户确认 Daemon 已启动且 Chat Completions 端点已启用；设置中「一键应用本地模型」可写入 Ollama 并设为默认，减少 404。

### 11.2 痛点

- 主界面需区分「已完成向导」与「未完成向导」：未完成时引导回到阶段一或阶段二，避免空白对话区。
- 设置修改后（如重装 openclaw、更换模型）可能影响当前对话或连接，需提示或提供「重新检测/重连」入口。
- **404 常见原因**：Daemon 未启动，或未在配置中启用 `gateway.http.endpoints.chatCompletions.enabled`；未配置默认模型时也可能导致网关行为异常，可通过「一键应用本地模型」补齐。

### 11.3 注意点

- 设置入口**位置**：主界面为**发送按钮旁**，与 00 文档 1.1 一致；主界面与设置页均可进入「OpenClaw 配置」页。

## 十二、必备设置菜单（贯穿）

### 12.1 执行方案

- **入口**：**阶段一**安装界面**右上角**常驻「设置」；**阶段三**主界面**输入框旁（发送按钮旁）**「设置」。**安装向导仅首次可见**：完成过一次向导后，从设置「返回」或「保存」进入**主界面**而非带「开始安装」的首页；设置页对已完成用户展示说明并按钮文案「返回主界面」。
- **必备项**：  
  - **OpenClaw 基础配置**：按钮「打开 OpenClaw 配置页」进入阶段二表单（模型来源、API Key、默认模型、网关端口等）。  
  - **一键应用本地模型**：下拉选择 Ollama 模型 + 按钮「一键应用本地模型」；执行：写入 `models.providers["ollama-local"]`、将默认模型设为该 Ollama、`modelSource: local`，可避免对话 404；需先启动 Ollama 并拉取模型。  
  - **Node 环境地址**：输入框或路径选择，展示当前检测到的路径；保存后所有依赖 Node/npm 的检测与安装均使用该路径（或该路径下的 node/npm）。  
  - **npm 源地址**：展示当前 registry；下拉或输入框，预设（官方、npmmirror 等）+ 自定义；保存时写 npm config set registry。  
  - **OpenClaw 包**：版本选择（如 latest）、「重新安装全局包」按钮、「打开配置目录」、Daemon 服务管理（安装并启动 Daemon）。
- **可选**：诊断日志导出、代理配置等。
- **与引导联动**：在设置中修改任意一项后，再次进入安装引导或点击「重新检测」时，功能检测与各步骤执行必须使用最新设置。

### 12.2 痛点

- **设置持久化**：Node 路径、registry 等需持久化到本地（如 electron-store 或应用配置目录），应用重启后仍生效；需区分「未设置」与「已清空」。
- **校验**：Node 路径若用户填错，保存后会导致后续检测与安装全部失败；可在保存时做一次简单校验（如 spawn 该路径下的 node -v），失败则提示「无法使用该路径」。
- **多语言与文案**：设置项说明需清晰（如「npm 源地址用于全局安装时的下载源，修改后仅影响本应用触发的安装」），避免用户误解作用范围。

### 12.3 风险点

- **设置与检测不同步**：修改设置后若未重新跑功能检测，引导页仍显示旧状态；应在保存设置后自动触发一次「轻量检测」或提示「请点击重新检测以更新状态」。
- **敏感信息**：若设置中保存了 API Key 或 token，需加密或仅存引用，避免配置文件泄露。
- **跨平台路径**：Node 路径在 Windows 上可能带盘符与反斜杠，在 Mac/Linux 为绝对路径；保存与读取时需统一、且 spawn 时正确传入。

### 12.4 注意点

- 设置页与安装引导共用同一套「功能检测服务」与「执行器」；避免两处逻辑重复或不一致。
- 「重新安装 OpenClaw」等破坏性操作需二次确认（如弹窗「将覆盖当前全局安装的 openclaw，是否继续？」）。
- 文档索引中可增加「设置说明」链接，说明各设置项含义与推荐值（如中国用户推荐 npmmirror）。

## 十四、GUI 桌面 Agent 完整方案（多步任务版）

> 本节用于替代“只做最小实现”的思路，目标是让 GUI 能力从「单步命令」升级到「可规划、可执行、可验证、可审计」的任务系统。

### 14.1 目标与非目标

- 目标：
  - 支持一句话触发多步任务（如「打开微信→搜索联系人→发送消息」）。
  - 每一步有前置条件、后置验证、失败重试、可中断。
  - 全链路可审计（任务级 + 步骤级），可导出与复盘。
- 非目标（当前版本不做）：
  - 跨平台一次性打满所有能力（先以 macOS 为主）。
  - 直接自动化高风险动作（支付、删库、凭据输入）默认放行。

### 14.2 分层架构

1. **Intent/Planner（规划层）**
   - 输入自然语言，输出结构化计划 `GuiTaskPlan`。
   - 计划包含：`taskId`、`steps[]`、`risk`、`requiresConfirmation`、`budget`。
2. **Policy/Guardrail（策略层）**
   - 基于 `guiEnabled`、应用白名单、动作白名单、风险等级进行准入决策。
   - 支持任务级预算：最大步数、最长耗时、最大重试次数。
3. **Perception（感知层）**
   - 前台应用识别、截图、OCR 文本锚点定位（后续可加图标模板匹配）。
   - 为步骤执行提供“是否已达到目标状态”的验证依据。
4. **Action Runtime（执行层）**
   - 统一动作接口：`open_app`、`focus_window`、`key_chord`、`type_text`、`press_key`、`click_text_anchor`、`wait_for_text`。
   - 每步支持重试、超时、失败分类与恢复分支。
5. **Audit/Replay（审计回放层）**
   - 记录任务与步骤日志、错误码、耗时、截图路径。
   - 支持导出报告与“仅重试失败步骤”。

### 14.3 关键数据结构（建议）

- `GuiTaskPlan`
  - `id`, `userIntent`, `risk`, `requiresConfirmation`, `steps[]`, `createdAt`
- `GuiStep`
  - `id`, `action`, `args`, `preconditions`, `postconditions`, `timeoutMs`, `maxRetries`
- `GuiStepResult`
  - `status`, `errorCode`, `message`, `attempt`, `durationMs`, `artifacts`
- `GuiTaskRun`
  - `taskId`, `state`, `currentStep`, `startedAt`, `endedAt`, `summary`

### 14.4 风险与控制策略

- 风险分级：
  - `low`：打开应用、窗口切换、普通输入
  - `medium`：系统设置导航、批量输入、跨应用复制粘贴
  - `high`：删除/覆盖/支付/账号凭据相关
- 默认策略：
  - `low` 自动执行；
  - `medium` 按策略可自动或需确认；
  - `high` 默认确认，未确认即阻断。
- 硬阻断：
  - 涉及支付、密码、验证码、密钥、隐私导出时默认阻断并提示手动。

### 14.5 分阶段实施（完整路线）

1. **Phase 1（当前要做）— 编排骨架 + 多步执行器**
   - 建立 `GuiTaskPlan` 与 `GuiStepRunner`；
   - 将现有 `open_app/hotkey/type_text` 接入 step 执行；
   - 回执升级为“任务进度回执（step x/n）”。
2. **Phase 2 — 感知闭环**
   - 接入截图/OCR；
   - 新增 `wait_for_text` 与 `click_text_anchor`；
   - 每步执行后进行后置验证。
3. **Phase 3 — 人机协作 + 安全增强**
   - 执行前展示计划摘要；
   - 高风险步骤逐条确认；
   - 支持暂停/继续/终止任务。
4. **Phase 4 — 稳定性与场景模板**
   - 场景模板（微信发消息、浏览器检索、文件归档）；
   - 异常回归集与成功率指标看板。

### 14.6 验收标准（建议）

- 功能：
  - 多步任务可按顺序执行，失败可定位到具体步骤。
- 稳定性：
  - 标准分辨率 + 中文界面下，核心用例成功率达到目标阈值（后续设定）。
- 安全：
  - 高风险误执行率为 0；
  - 每次阻断都有明确 reason 与 nextAction。
- 可观测：
  - 审计日志可按任务 ID 回放到步骤级细节。
